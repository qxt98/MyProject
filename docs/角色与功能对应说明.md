## 4. 分析结论与修正

### 4.1 不一致项

- **药剂科人员（PHARMACIST）**  
  需求明确包含「**采购申请**」，即应能进入采购审批模块并提交/查询采购申请。当前后端与前端均未对 PHARMACIST 开放采购审批，与需求不符。

### 4.2 已做修正

- **药剂科人员（PHARMACIST）**：已在后端与前端为其开放**采购审批**（与需求「采购申请」一致），使其能提交并查询采购申请；审批操作仍由审核人员（REVIEWER）负责。

### 4.3 其他说明

- **护士（NURSE）**：需求变更为「护士账号不能有库存管理，处方审核的模块的权限」；已移除护士对库存管理和处方审核的所有权限。
- **药品信息**：PURCHASER、REVIEWER 仅需在采购/业务中选药品，故仅后端 GET 药品开放，前端不展示「药品信息」菜单，符合设计。

## 7. 设计合理性分析与改进意见

### 7.1 当前设计的优点

- **角色与需求基本对齐**：系统管理员、药剂科、采购、审核、医生、护士与需求文档 2.2 一致，且已按需求为药剂科开放采购申请。
- **前后端双重控制**：后端接口按角色拦截，前端按角色控制菜单与路由，避免未授权入口。
- **药品只读分离**：采购/审核员仅 GET 药品用于下拉，不开放药品管理菜单，符合最小可见原则。
- **实现简单**：基于角色的模块级权限，配置集中、易于理解和维护。

### 7.2 存在的问题与风险

| 问题 | 说明 | 风险等级 | 状态 |
|------|------|----------|------|
| **采购/处方未按“动作”细分** | 采购审批、处方审核的“审批”动作应与“申请/开立”分离。 | 中 | **已落实**（@PreAuthorize + 前端按钮按角色隐藏） |
| **患者/访客未落地** | 需求中“患者/访客”使用用药指导。 | 低 | **已落实**（POST /guide/ask permitAll） |
| **护士处方查询未开放** | 需求为“处方查询等（按实际授权）”。 | 低 | **已落实**（GET 处方 + NURSE，前端护士只读） |
| **文档与代码可能不同步** | 权限变更时需同时改 SecurityConfig、前端 ROUTE_ROLES、本文档三处，易遗漏。 | 低 | 建议保留约定 |

### 7.3 改进建议

#### （1）采购/处方按“动作”做接口级权限（推荐）【已实现】

- **采购**  
  - **提交/查询**（GET /purchase、GET /purchase/{id}、POST /purchase）：ADMIN、PHARMACIST、PURCHASER、REVIEWER。  
  - **审批**（POST /purchase/{id}/approve、POST /purchase/{id}/reject）：仅 ADMIN、REVIEWER（`@PreAuthorize`）。  
- **处方**  
  - **开立/提交/查看**（GET、POST 创建、POST 提交等）：ADMIN、REVIEWER、DOCTOR；GET 另开放 NURSE。  
  - **审核通过/驳回**（POST approve/reject）：仅 ADMIN、REVIEWER（`@PreAuthorize`）。  

已实现：PurchaseRequestController、PrescriptionController 的 approve/reject 方法加 `@PreAuthorize("hasAnyRole('ADMIN', 'REVIEWER')")`；前端采购/处方页按角色隐藏“通过”“驳回”按钮。单元测试见 `PurchaseRequestControllerSecurityTest`、`PrescriptionControllerSecurityTest`。

#### （2）权限配置单一数据源（可选）

- 将“角色–可访问路径/操作”集中到一张表或一份配置（如 YAML/数据库），后端与前端均从该配置推导权限，文档由配置自动生成或与配置强一致，减少三处不一致的可能。

#### （3）患者/访客用药指导（按需求取舍）【已实现】

- 已实现：POST `/guide/ask`、`/api/guide/ask` 在 SecurityConfig 中设为 `permitAll()`，未登录患者/访客可匿名咨询。单元测试见 `GuideControllerSecurityTest`。

#### （4）护士处方权限调整（按最新需求）

- 最新需求：护士账号**不能**访问库存管理与处方审核模块。已在 SecurityConfig 中移除 NURSE 对 `/stock/**` 与 `/prescription/**` 的权限，并在前端 ROUTE_ROLES 中移除护士对 `/stock`、`/prescription` 路由的访问权限。

#### （5）文档与代码同步约定

- 约定：凡修改角色权限，必须同步更新（1）后端 SecurityConfig、（2）前端 ROUTE_ROLES、（3）本文档“角色与功能矩阵”和“后端接口速查”。  
- 可选：在 CI 或本地脚本中，从 SecurityConfig 或开放 API 列表生成文档片段，减少手写表格带来的偏差。

### 7.4 总结

| 维度 | 评价 | 说明 |
|------|------|------|
| 与需求一致性 | 良好 | 已按需求开放药剂科采购申请、护士处方只读、患者/访客用药指导。 |
| 权限粒度 | 已改进 | 采购/处方已按“申请 vs 审批”做接口级区分（@PreAuthorize + 前端按钮）。 |
| 可维护性 | 良好 | 配置集中；建议通过单一数据源或自动生成文档降低遗漏。 |
| 扩展性 | 良好 | 新增角色或模块时，在现有 RBAC 上扩展即可。 |

**结论**：7.3 改进建议（1）（3）（4）已落实；采购与处方按动作细分权限，患者/访客可匿名用药咨询，护士可查处方。单元测试见 `backend/src/test/java/com/smartpharma/controller/` 下 `PurchaseRequestControllerSecurityTest`、`PrescriptionControllerSecurityTest`、`GuideControllerSecurityTest`。

### 运行方式

- **要求**：JDK 17+、Maven（与后端一致）。若本机 JDK 版本较低，可用 Docker 运行测试。
- **运行全部测试**：在 `backend` 目录执行 `mvn test`。
- **仅运行上述权限测试**：
  ```bash
  cd backend
  mvn test -Dtest=PurchaseRequestControllerSecurityTest,PrescriptionControllerSecurityTest,GuideControllerSecurityTest
  ```
- **使用 Docker 运行全部测试**（不依赖本机 JDK）：
  ```bash
  docker run --rm -v "<项目路径>/backend:/app" -w /app maven:3.9-eclipse-temurin-17 mvn test -B
  ```
  将 `<项目路径>` 替换为实际路径（如 Windows 下 `e:/workspace/smart-pharma`）。

### 运行项目（Docker）

在项目根目录执行：

```bash
docker compose up -d --build
```

- 前端：http://localhost:3000  
- 后端 API：http://localhost:8080/api  
- 默认管理员：用户名 `admin`，密码 `123456`  
- **患者用药指导**：患者无需登录即可使用，直接访问 http://localhost:3000/guide 或从登录页点击「患者用药指导（无需登录）」进入。回复支持接入 DeepSeek 等 AI（通过环境变量 `GUIDE_AI_API_KEY` 或 `application.yml` 配置），未配置时使用规则兜底；详见开发与部署说明「患者用药指导 AI 配置」。

更多运行方式见 [README.md](../README.md) 的「开发与测试」一节。
