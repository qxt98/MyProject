# 流式回答（SSE + 边收边展示）与前端交互 — 学习文档

## 文档说明

本文档面向希望理解「患者用药指导」中**流式回答**实现原理的同学，涵盖：为何使用流式、SSE 是什么、后端如何推送、前端如何消费与展示，以及与本项目代码的对应关系。可作为毕业设计答辩的讲解材料或二次开发的参考。

---

## 1. 背景与目标

### 1.1 问题

- **一次性请求**：前端提交问题后，后端等 AI 生成**整段回答**再一次性返回。大模型生成可能需 10～30 秒，用户长时间白屏或转圈，容易误以为超时或卡死。
- **超时**：HTTP 或前端请求若设了较短超时（如 10 秒），在整段生成前就可能被判定为超时而失败。

### 1.2 思路：流式回答

- **边生成边返回**：后端在 AI 每生成一小段（如几个字）就立刻推送给前端，前端**边收边展示**。
- **效果**：用户很快看到首字出现，随后内容逐步增加，体验类似「打字机」；无需等整段生成完毕，也降低超时概率。

### 1.3 技术选型：SSE（Server-Sent Events）

- **SSE**：基于 HTTP 的**服务端主动推送**协议，一次 HTTP 请求、响应体为长连接流，服务端可多次发送「事件」。
- **与本项目**：后端用 Spring 的 `SseEmitter` 先发 `start` 建立流，再发 content、disclaimer、remainingAsks、error；前端用 `fetch` 拿到 `ReadableStream`，按 SSE 格式解析后更新界面。
- **与 WebSocket 对比**：SSE 仅服务端→客户端单向，实现简单、无需额外端口，适合「一问一答、服务端持续推」的场景；WebSocket 适合双向、高频交互。

## 2. 核心概念速览


| 概念                | 说明                                                         |
| ----------------- | ---------------------------------------------------------- |
| **流式（Streaming）** | 数据分多次、按序从服务端发往客户端，而不是等全部生成完再一次性返回。                         |
| **边收边展示**         | 前端每收到一段数据就追加到界面（如 `answer += chunk`），用户实时看到内容增长。           |
| **SSE 事件**        | 每条事件由多行文本组成：`event: 事件名`、`data: 数据`，以双换行 `\n\n` 结束。        |
| **SseEmitter**    | Spring MVC 提供的类，用于在 Controller 中向当前 HTTP 响应写入 SSE 事件并保持连接。 |


---

## 3. 整体数据流

```
用户点击「提交咨询」
       ↓
前端 fetch POST /api/guide/ask/stream，body: { question }
       ↓
后端校验（长度、限流等）→ 创建 SseEmitter，立即 return emitter（连接保持）
       ↓
后端异步线程：先 emitter.send("start", "ok") 建立流，再调用 streamAsk
       ↓
每解析到一段 content → emitter.send(event("content", chunk)) → 前端收到
       ↓
前端 onContent(chunk) → streamedAnswer.value += chunk → 界面即时更新
       ↓
AI 流结束 → 后端发送 disclaimer、remainingAsks → emitter.complete()
       ↓
前端 onDisclaimer / onRemainingAsks / onDone → 收起加载态、展示免责与剩余次数
异常时后端发送 event("error", msg)，前端 onError；流式失败时前端回退调用一次性接口 askGuide
```

## 4. 后端实现要点

### 4.1 接口形态

- **路径**：`POST /api/guide/ask/stream`
- **响应类型**：`Content-Type: text/event-stream`（SSE 规范要求）
- **返回类型**：`SseEmitter`。Controller 创建 emitter 后**立即 return**，实际写入在异步线程中完成，这样不会占用请求线程长时间阻塞。

### 4.2 关键代码位置与逻辑


| 文件                                                    | 作用                                                                                                                                                                                                                             |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `GuideController.askStream()`                         | 校验参数与限流；创建 `SseEmitter(60_000L)`（60 秒超时）；用 `CompletableFuture.runAsync` 在后台先发送 `event("start", "ok")` 建立流，再执行 `guideAiService.streamAsk(...)`；异常时发送 `event("error", msg)` 后 `completeWithError`；立即 return emitter。             |
| `GuideAiService.streamAsk(question, onChunk, onDone)` | 若未启用 AI，则规则兜底：一次 `onChunk(全文)` + `onDone()`。若启用 AI，则请求 AI 时带 `stream: true`，用 `RestTemplate.execute` 拿到响应流，按行解析 SSE，从每行 `data: {...}` 中取 `choices[0].delta.content`，调用 `onChunk(content)`；遇到 `data: [DONE]` 或流结束调用 `onDone()`。 |
| Controller 内 onChunk / onDone                         | onChunk：`emitter.send(SseEmitter.event().name("content").data(chunk, MediaType.TEXT_PLAIN))`；onDone：发送 `disclaimer`、可选 `remainingAsks`，再 `emitter.complete()`。                                                                 |


### 4.3 事件类型约定（前后端一致）


| event 名         | 含义                  | 前端处理                                |
| --------------- | ------------------- | ----------------------------------- |
| `start`         | 建立流（立即发送）           | 忽略，仅用于确认连接                          |
| `content`       | 一段回复正文              | 追加到 `streamedAnswer`                |
| `message`       | SSE 默认事件名，同 content | 同 content，追加到 `streamedAnswer`      |
| `disclaimer`    | 免责声明                | 写入 `result.disclaimer`              |
| `remainingAsks` | 剩余提问次数（匿名）          | 更新 `remainingAsks`（解析时 event 转小写匹配） |
| `error`         | 错误信息（如限流、异常）        | 展示错误、清空 `streamedAnswer`            |


### 4.4 为何用异步线程

- `streamAsk` 内会**阻塞**地读 AI 的响应流（如 `reader.readLine()`），若在主请求线程执行，该线程会一直被占用直到流结束。
- 用 `CompletableFuture.runAsync` 在池中线程执行，Controller 方法可立即返回，Tomcat 不会认为请求未完成而回收连接；同时该线程专门负责读流、回调 onChunk/onDone 并向 emitter 写数据。

## 5. 前端实现要点

### 5.1 为何用 fetch 而不是 Axios

- 本项目常规接口用 **Axios**，响应由拦截器解析为 JSON，**一次性**得到 `res.data`。
- 流式接口需要**持续读取响应体**，使用 **fetch** 的 `response.body`（`ReadableStream`），通过 `reader.read()` 循环取块（chunk），再按 SSE 格式解析。Axios 默认不会暴露这种「未消费完的流」，故流式用 fetch 更直接。

### 5.2 关键代码位置与逻辑


| 文件                                               | 作用                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `frontend/src/api/guide.js` 的 `askGuideStream()` | 使用 `fetch(..., { method: 'POST', body: JSON.stringify({ question }) })`，请求头含 `Content-Type` 与可选的 `Authorization`；取 `res.body.getReader()`；用 `TextDecoder('utf-8')` 解码；buffer 统一 `\r\n`/`\r` 为 `\n`，按 `/\n\n+/` 拆事件；每行 `trim()` 后识别 `event:`（`slice(6).trim().toLowerCase()`）与 `data:`（`slice(5).trimStart()`），支持多行 data 用 `\n` 拼接；对 data 做 `unquoteData()` 处理 JSON 引号；event 为 `content` 或 `message` 且 data 非空时调用 onContent，无 event 但有 data 时也调用 onContent；最后 onDone()。流式失败时由 Guide.vue 的 catch 回退调用一次性接口 `askGuide`。 |
| `frontend/src/views/Guide.vue`                   | 使用独立 ref `streamedAnswer` 专门存放流式正文；提交时 `result = { answer: '', disclaimer: '' }`、`streamedAnswer = ''`、`loading = true`；onContent：`streamedAnswer.value += chunk`；onDisclaimer：`result.value.disclaimer = text`；回复区显示条件为 `showReplyCard`（`result !== null || streamedAnswer !== ''`），正文绑定 `{{ streamedAnswer }}`，loading 时显示光标 `▌`；流式抛错时 catch 内调用 `askGuide` 并设 `streamedAnswer = res.data.answer`。                                                                                                                 |


### 5.3 SSE 解析要点

- 服务端每发一条事件，格式类似：
  ```text
  event: content
  data: 你好

  ```
  （注意 `data:` 后可有空格，多条 `data:` 行会按 `\n` 拼接；结尾为双换行；支持 `event:content` 无空格。）
- 前端收到**二进制块**，先 `decoder.decode(value, { stream: true })` 再拼到 buffer；`buffer.replace(/\r\n/g,'\n').replace(/\r/g,'\n')` 统一换行后，用 `buffer.split(/\n\n+/)` 得到完整事件（未完成的一段用 `parts.pop()` 留回 buffer）。
- 每行 `trim()` 避免 BOM/首尾空格；event 名转小写后匹配（如 `remainingAsks` → `remainingasks`）；若 data 为 JSON 引号字符串（如 `"你好"`），用 `unquoteData()` 解析后传入回调；仅当 data 非空时对 content/message 调用 onContent。

### 5.4 与 Vue 响应式的配合

- 流式正文用**单独 ref** `streamedAnswer`（`ref('')`），onContent 中执行 `streamedAnswer.value += chunk`，模板绑定 `{{ streamedAnswer }}`，保证每次追加都触发视图更新。
- 免责声明仍用 `result.disclaimer`，onDisclaimer 时写 `result.value.disclaimer = text`。
- 回复卡片是否展示由 `showReplyCard`（`result !== null \|\| streamedAnswer !== ''`）控制；加载态用 `loading`，onDone 或 onError 时设为 false，模板中 `v-if="loading"` 显示光标。

## 6. 关键代码片段说明

### 6.1 后端：先发 start 再发送 content 事件

```java
// GuideController 内，异步任务中
emitter.send(SseEmitter.event().name("start").data("ok", MediaType.TEXT_PLAIN));
guideAiService.streamAsk(
    questionFinal,
    chunk -> {
        try {
            emitter.send(SseEmitter.event().name("content").data(chunk, MediaType.TEXT_PLAIN));
        } catch (IOException e) { throw new RuntimeException(e); }
    },
    () -> {
        try {
            emitter.send(SseEmitter.event().name("disclaimer").data(DISCLAIMER, MediaType.TEXT_PLAIN));
            if (anonymousFinal) emitter.send(SseEmitter.event().name("remainingAsks").data(...));
            emitter.complete();
        } catch (IOException e) { emitter.completeWithError(e); }
    }
);
// 异常时：emitter.send(SseEmitter.event().name("error").data(e.getMessage(), ...)); emitter.completeWithError(e);
```

- 先发 `start` 便于前端/代理确认流已建立；`SseEmitter.event().name("content").data(chunk, MediaType.TEXT_PLAIN)` 生成符合 SSE 的 `event: content` 与 `data: ...`，通过当前响应流发往客户端。

### 6.2 后端：解析 AI 流并回调 onChunk

```java
// GuideAiServiceImpl 内，response 为 AI 接口的响应流
while ((line = reader.readLine()) != null) {
    if (!line.startsWith("data: ")) continue;
    String data = line.substring(6).trim();
    if ("[DONE]".equals(data)) break;
    // 解析 JSON，取 choices[0].delta.content
    String content = delta.get("content").asText("");
    if (!content.isEmpty()) onChunk.accept(content);
}
onDone.run();
```

- OpenAI 兼容接口在 `stream: true` 时返回的也是 SSE 格式，每行 `data: {...}` 对应模型输出的一小段增量（delta）。

### 6.3 前端：读流并解析 SSE

```javascript
// guide.js 中
const decoder = new TextDecoder('utf-8')
let buffer = ''
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  buffer += decoder.decode(value, { stream: true })
  buffer = buffer.replace(/\r\n/g, '\n').replace(/\r/g, '\n')
  const parts = buffer.split(/\n\n+/)
  buffer = parts.pop() || ''
  for (const part of parts) {
    let event = '', dataLines = []
    for (const line of part.split('\n')) {
      const trimmed = line.trim()
      if (trimmed.startsWith('event:')) event = trimmed.slice(6).trim().toLowerCase()
      else if (trimmed.startsWith('data:')) dataLines.push(trimmed.slice(5).trimStart())
    }
    const rawData = dataLines.join('\n').trim()
    const data = rawData !== undefined ? unquoteData(rawData) : rawData
    if (event === 'content' && data) onContent?.(data)
    else if (event === 'message' && data) onContent?.(data)
    else if (event === 'disclaimer' && data) onDisclaimer?.(data)
    else if (event === 'remainingasks' && data) onRemainingAsks?.(parseInt(data, 10))
    else if (event === 'error' && data) onError?.(data)
    else if (data && !event) onContent?.(data)
  }
}
onDone?.()
```

- `TextDecoder('utf-8')` 与 `stream: true` 保证多字节字符不在块边界截断导致乱码。
- 统一 `\r\n`/`\r` 为 `\n`，用 `/\n\n+/` 拆分事件；未完成的一段用 `parts.pop()` 留回 buffer。
- `unquoteData(data)` 处理 data 为 JSON 引号字符串（如 `"你好"`）时去掉引号再交给回调。

### 6.4 前端：边收边展示与流式失败回退

```javascript
// Guide.vue 中
streamedAnswer.value = ''
result.value = { answer: '', disclaimer: '' }
await askGuideStream(q, {
  onContent: (chunk) => { streamedAnswer.value += chunk },
  onDisclaimer: (text) => { if (result.value) result.value.disclaimer = text },
  onDone: () => { loading.value = false },
  // onError 时清空 streamedAnswer、result，展示 limitError 或 alert
})
```

- 流式正文单独用 `streamedAnswer`（`ref('')`），onContent 时 `streamedAnswer.value += chunk`，模板绑定 `{{ streamedAnswer }}`，保证每次追加都触发视图更新；免责声明用 `result.disclaimer`。
- 回复区显示条件为 `showReplyCard`（`result !== null || streamedAnswer !== ''`），loading 时显示光标 `▌`。
- 若 `askGuideStream` 抛错（如网络异常），在 catch 中回退调用一次性接口 `askGuide(q)`，用返回的 `res.data.answer` 赋给 `streamedAnswer`，保证仍有回复可看。

## 7. 时序与状态


| 时间点   | 后端                                                                 | 前端                                                                                          |
| ----- | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------- |
| T0    | 收到 POST，校验通过，创建 SseEmitter，提交异步任务，return emitter                   | fetch 发出，得到 response，response.body 可读                                                       |
| T0+   | 异步线程先 emitter.send("start", "ok")，再调用 streamAsk                    | 解析到 start 事件（可忽略），流已建立                                                                      |
| T1～Tk | 每有 content 就 emitter.send("content", chunk)                        | 解析出 content/message 事件，onContent(chunk) → streamedAnswer += chunk，界面刷新                      |
| Tk+1  | 发送 disclaimer、remainingAsks，emitter.complete()                     | 解析到对应事件，更新 disclaimer、remainingAsks；流结束 onDone()，loading = false                            |
| 异常    | 校验/限流失败发 error 后 complete；streamAsk 异常先发 error 再 completeWithError | onError(msg) 清空 streamedAnswer/result；若 fetch 抛错则 catch 中回退 askGuide，用一次性结果填 streamedAnswer |

## 8. 扩展与注意点

- **超时**：SseEmitter(60_000L) 表示 60 秒内若无新数据或完成，连接可被释放；可根据需要调整。前端 fetch 默认无超时，长时间无数据可能一直挂起，可按需加 AbortController。
- **编码**：后端发送与前端解码均使用 UTF-8（前端显式 `new TextDecoder('utf-8')`），避免中文乱码。
- **错误与重试**：错误通过 `event: error` 或 HTTP 4xx/5xx 传递；前端 onError 时清空 `streamedAnswer` 与 `result`；若流式请求本身抛错（如网络），catch 中回退调用一次性接口 `askGuide`，保证有回复可展示。
- **规则兜底**：当 AI 未配置或调用失败时，后端仍通过同一套流式接口推送，整段内容一次 onChunk，前端逻辑无需区分。
- **SSE 兼容**：前端对 event 名做 toLowerCase 匹配（如 remainingAsks → remainingasks）；支持无 event 行、仅 `data: xxx` 时当作 content；对 data 做 `unquoteData` 以兼容 JSON 引号包裹的字符串。
