## 1. 引言

### 1.1 编写目的

本需求文档旨在明确“药品信息智能化管理系统”的功能需求与非功能需求，为设计、开发、测试与验收提供依据，并作为与指导教师、答辩评委沟通的基准文档。

### 1.2 项目背景与范围

- **背景**：医院药品管理存在手工记录效率低、易出错等问题，需通过信息化与适度智能化实现药品信息的规范化管理、库存的实时监测与调控、采购与处方流程的规范化及患者用药指导。
- **范围**：本系统覆盖药品基础信息管理、库存管理、采购审批、处方审核、患者用药指导及系统管理六大功能域，面向院内药剂科、采购人员、审核人员、医生、护士及患者（或访客）等用户角色。

### 1.3 术语与缩略语

| 术语/缩略语 | 说明 |
|-------------|------|
| RBAC       | 基于角色的访问控制（Role-Based Access Control） |
| SPA        | 单页应用（Single Page Application） |
| API        | 应用程序接口（Application Programming Interface） |
| 处方审核   | 对医生开具的处方进行规范性审查与风险识别 |
| 配伍禁忌   | 两种或多种药物联用可能产生的不良相互作用 |

### 1.4 参考文献

- 毕业设计开题报告（基于 Spring Boot 的药品信息智能化管理系统的设计与实现）
- 项目设计说明（docs/设计说明.md）
- 《网络安全法》《个人信息保护法》等相关法规

## 2. 项目概述

### 2.1 系统目标

- 实现药品基础信息的数字化、规范化维护与查询。
- 实现药品库存的实时监测、入库/出库/退库管理及库存预警。
- 实现药品采购申请的提交与审批流程管理。
- 实现处方的规范化审查与用药风险提示。
- 为患者提供基于规则的用药咨询与指导。
- 实现用户、角色、权限及操作日志等系统管理功能，保障系统安全与可审计。

### 2.2 用户与角色

| 角色         | 说明 | 主要使用功能 |
|--------------|------|----------------|
| 系统管理员   | 负责系统配置、用户与权限管理 | 系统管理、各模块的维护与查询 |
| 药剂科人员   | 负责药品与库存日常管理     | 药品信息管理、库存管理、采购申请 |
| 采购人员     | 负责提交采购申请、跟进审批 | 采购审批（申请提交与查询） |
| 审核人员     | 负责采购与处方审批         | 采购审批（审批操作）、处方审核 |
| 医生         | 开具处方                   | 处方录入与查询、查看药品列表以选择处方药品 |
| 护士         | 执行领药、发药等           | 库存出库、处方查询等（按实际授权） |
| 患者/访客    | 获取用药建议               | 患者用药指导 |

不同角色通过 RBAC 获得不同菜单与接口访问权限，避免越权操作。

## 3. 功能需求

### 3.1 药品信息管理模块

**模块描述**：对系统内药品基础信息进行规范化维护与精细化管理。

**功能需求列表**：

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-Drug-01 | 系统应支持按药品名称、类别等条件查询药品列表，并支持分页展示。 | 高 |
| F-Drug-02 | 系统应支持新增药品，录入项至少包括：药品名称、药品类别、剂型规格、适应症描述、生产厂家、供应商信息、采购价、销售价、药品图片（可选）。 | 高 |
| F-Drug-03 | 系统应支持对已有药品信息进行修改，确保数据一致性与审计可追溯。 | 高 |
| F-Drug-04 | 系统应支持删除或停用药品记录；删除时需考虑是否被库存、处方等引用，必要时采用逻辑删除。 | 高 |
| F-Drug-05 | 药品名称、类别、剂型规格、适应症、厂家、供应商、价格等关键属性应能完整展示与编辑。 | 高 |
| F-Drug-06 | （可选）支持药品信息的批量导入、导出，便于与现有数据或外部系统对接。 | 中 |

### 3.2 库存管理模块

**模块描述**：对药品库存状态进行实时监测与动态调控，保障供应稳定性。

**功能需求列表**：

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-Stock-01 | 系统应支持药品入库登记，录入项至少包括：药品、数量、批次号、生产日期、有效期、备注。入库后自动更新或生成对应库存记录。 | 高 |
| F-Stock-02 | 系统应支持药品出库登记，选择药品（及批次）、数量，出库后扣减对应库存，并记录出库流水。 | 高 |
| F-Stock-03 | 系统应支持退库操作，将已出库药品退回库存，并更新库存数量与流水。 | 高 |
| F-Stock-04 | 系统应支持按药品、批次、时间范围等条件查询库存明细及入库/出库/退库流水。 | 高 |
| F-Stock-05 | 系统应支持库存预警：当某药品库存数量低于设定阈值时进行低库存预警；当某批次药品有效期临近（如 6 个月内）时进行近效期预警。预警阈值可通过系统参数配置。 | 高 |
| F-Stock-06 | 库存变动应由系统自动计算并维护，保证数量一致性与事务完整性。 | 高 |

### 3.3 采购审批模块

**模块描述**：实现药品采购流程的规范化管理与审批操作。

**功能需求列表**：

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-Pur-01 | 系统应支持提交药品采购申请，申请内容至少包括：药品名称、采购数量、供应商信息、预算金额、采购理由、申请人。 | 高 |
| F-Pur-02 | 采购申请提交后应进入“待审批”状态，并可由具备审批权限的人员进行审批或驳回。 | 高 |
| F-Pur-03 | 审批通过后，系统应记录审批人、审批时间；可选：自动生成采购单或触发后续流程。 | 高 |
| F-Pur-04 | 系统应支持按状态、时间、药品等条件查询采购申请列表及详情。 | 高 |
| F-Pur-05 | 驳回时需支持填写驳回原因，并通知申请人（或在其列表中可见）。 | 中 |

### 3.4 处方审核模块

**模块描述**：对医疗机构内部处方进行规范化审查与用药风险控制。

**功能需求列表**：

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-Rx-01 | 系统应支持处方信息的录入或导入，处方至少包含：患者信息、**患者病情信息（主诉、诊断等，供审核参考）**、处方明细（药品、剂量、给药途径、用药时长/频次等）。 | 高 |
| F-Rx-02 | 系统应对处方内容与药品数据库进行比对，校验药品名称、剂量规格、给药途径等是否合法、合理。 | 高 |
| F-Rx-03 | 系统应能识别并提示潜在风险：重复用药、剂量异常、过敏风险、配伍禁忌或不合理联用等。 | 高 |
| F-Rx-04 | 审核人员可对处方进行“通过”或“不通过”操作，并可填写审核意见；系统记录审核人与审核时间。 | 高 |
| F-Rx-05 | 系统应支持按患者、医生、时间、状态等条件查询处方列表及审核结果。 | 高 |
| F-Rx-06 | 系统应支持在处方中录入与展示患者病情信息，在新建处方、处方列表、处方详情及处方明细视图中均可填写或查看，供审核人员参考。 | 高 |

### 3.5 患者用药指导模块

**模块描述**：通过智能化交互方式向用户提供用药咨询与指导，提升用药安全与依从性。本模块**对患者/访客开放**，无需登录即可使用：前端提供公开页（/guide），登录页有「患者用药指导（无需登录）」入口；后端咨询接口（POST /guide/ask）允许匿名访问。未登录用户存在提问次数限制与安全约束，已登录用户不限。

**功能需求列表**：

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-Guide-01 | 系统应提供用药指导入口，用户可输入症状、既往病史、当前所用药品等文本信息。 | 高 |
| F-Guide-02 | 系统应对用户输入进行解析，结合药品数据库、临床用药规范及药物相互作用规则，生成用药建议或风险提示。 | 高 |
| F-Guide-03 | 系统应以自然语言或结构化方式回复用户，便于理解；已支持规则引擎与 AI（如 DeepSeek）双模式：配置 API Key 时使用 AI，未配置时使用规则兜底。 | 高 |
| F-Guide-04 | 回复内容应避免替代医嘱，并可在界面注明“仅供参考，请遵医嘱”等免责提示。 | 中 |
| F-Guide-05 | 未登录用户在同一时间窗口内（如 1 小时）按 IP 限制提问次数（如 3 次），超限时返回明确提示并引导登录或稍后再试；已登录用户不限制。 | 高 |
| F-Guide-06 | 系统应对提问内容做长度与格式校验（如最大字符数），防止滥用与超长输入；接口不向用户暴露内部错误细节。 | 中 |

### 3.6 系统管理模块

**模块描述**：对系统用户、角色权限、操作日志及基础参数进行统一管理。

**功能需求列表**：

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-Sys-01 | 系统应支持用户账号的增删改查，用户属性至少包括：用户名、密码、姓名、角色、启用/停用状态。 | 高 |
| F-Sys-02 | 系统应采用 RBAC 模型，支持角色与权限（菜单/接口）的配置，不同角色拥有不同功能访问范围。 | 高 |
| F-Sys-03 | 用户登录后仅能访问其角色所授权的菜单与接口，未授权访问应被拒绝并记录。 | 高 |
| F-Sys-04 | 系统应记录关键操作日志，至少包括：操作用户、操作时间、操作模块、操作类型、相关数据或摘要、IP 等，支持按条件查询。 | 高 |
| F-Sys-05 | 系统应支持部分业务参数的配置，如库存预警阈值、近效期天数、审批超时时间等，便于运维调整。 | 中 |
| F-Sys-06 | 用户密码应以加密方式存储，传输阶段采用安全协议（如 HTTPS）；敏感数据应脱敏展示。 | 高 |

## 4. 非功能需求

### 4.1 性能需求

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| NF-P-01 | 列表查询（分页）在常规数据量下（如单表万级）响应时间不超过 3 秒。 | 高 |
| NF-P-02 | 系统应支持多用户并发访问，关键接口在典型并发下保持稳定，不出现数据错乱。 | 高 |
| NF-P-03 | 采用数据库索引、查询优化及必要的缓存（如热点药品信息）以降低数据库压力、提高响应速度。 | 中 |

### 4.2 安全需求

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| NF-S-01 | 系统应实现身份认证（登录），并采用 Token（如 JWT）维持会话，防止未授权访问。 | 高 |
| NF-S-02 | 权限控制采用 RBAC，接口与菜单均按角色进行访问控制。 | 高 |
| NF-S-03 | 密码及敏感信息加密存储与传输；涉及患者信息时需符合《个人信息保护法》等要求。 | 高 |
| NF-S-04 | 对关键操作进行日志记录，支持安全审计与事后追溯。 | 高 |
| NF-S-05 | 对匿名开放接口（如患者用药指导）应做限流与输入校验，防止滥用与恶意请求；错误响应不暴露内部实现细节。 | 中 |

### 4.3 可用性与易用性

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| NF-U-01 | 界面应简洁清晰，主要操作有明确指引或提示，减少误操作。 | 高 |
| NF-U-02 | 关键业务操作（如删除、审批）应有二次确认或明确提示。 | 高 |
| NF-U-03 | 系统应具备一定的容错与友好错误提示，避免向用户暴露敏感技术细节。 | 中 |

### 4.4 兼容性与运行环境

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| NF-E-01 | 前端支持主流浏览器（如 Chrome、Edge、Firefox）的近期版本。 | 高 |
| NF-E-02 | 后端运行环境：JDK 17+、MySQL 8.0+；前端构建与运行需 Node.js 18+。 | 高 |

### 4.5 可维护性与可扩展性

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| NF-M-01 | 采用分层与模块化设计，便于后续功能扩展与维护。 | 中 |
| NF-M-02 | 关键业务逻辑应有单元测试或集成测试覆盖，保证回归质量。 | 中 |

## 5. 约束与假设

### 5.1 约束

- 系统需遵循国家与地方关于网络安全、个人信息保护、药品管理等相关法律法规。
- 技术栈限定：前端 Vue.js、后端 Spring Boot、数据库 MySQL，与开题报告一致。
- 部署环境为校内或院内局域网/单机环境，具体网络与服务器条件以实际为准。

### 5.2 假设

- 用户具备基本计算机操作能力，并接受简单培训或说明即可使用系统。
- 药品主数据、供应商信息、用药规范与配伍规则等由业务方提供或从权威数据源整理，本系统负责录入、维护与使用。
- 患者用药指导模块不替代医生诊断与处方，仅作为参考性信息展示。
- 初期版本以单机或单应用部署为主，高可用、分布式扩展为后续可选方向。
