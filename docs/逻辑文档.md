## 2. 药品信息管理逻辑

### 2.1 数据规则

- **药品名称**：必填，唯一性可由业务约定（本系统未强制唯一）。  
- **采购价、销售价**：数值型，可为空；若填写应为非负数。  
- **停用**：disabled=true 表示停用，列表默认不展示停用药品（includeDisabled=false）。

### 2.2 删除逻辑

- 当前为**物理删除**。若药品已被库存（stock）、处方明细（prescription_item）、采购申请（purchase_request）引用，删除前应在业务层校验或改为**逻辑删除**（仅将 disabled 置为 true），避免外键或业务不一致。

### 2.3 查询逻辑

- 列表支持：名称模糊匹配、类别精确匹配、是否包含停用、分页（page、size）。  
- 排序：按 id 降序，即最新创建的在前。

## 3. 库存管理逻辑

### 3.1 库存模型

- 库存粒度：**药品 + 批次**（drug_id + batch_no）唯一确定一条库存记录。  
- 同一药品可有多个批次（不同 batch_no），每个批次独立数量、生产日期、有效期。

### 3.2 入库逻辑

1. 写入**入库流水**（stock_in）：药品、批次、数量、生产日期、有效期、备注、创建时间。  
2. 查库存表是否存在该 drug_id + batch_no：  
   - **存在**：在原数量上增加本次入库数量，并更新生产日期、有效期（以本次为准）。  
   - **不存在**：新增一条库存记录，数量为本次入库数量。  
3. 同一事务内完成，保证流水与库存一致。

### 3.3 出库逻辑

1. 根据 drug_id + batch_no 查询库存记录。  
2. **校验**：库存数量 ≥ 出库数量，否则报错“库存不足”。  
3. 库存数量扣减，写**出库流水**（stock_out）。  
4. 同一事务内完成。

### 3.4 退库逻辑（需求 F-Stock-03）

- 业务上可理解为：将某次出库的全部或部分数量“退回”库存。  
- 实现方式之一：调用一次“入库”逻辑，备注中注明为退库并关联原出库单号（需在表结构或业务上支持）。当前版本若未单独实现退库接口，可在产品文档中说明为后续扩展。

### 3.5 预警逻辑

- **低库存**：库存表 quantity < 设定阈值（如 10），即列入低库存预警。  
- **近效期**：库存表 expiry_date ≤ 当前日期 + 设定天数（如 180 天），即列入近效期预警。  
- 阈值与天数可配置（系统参数或接口参数）。

## 4. 采购审批逻辑

### 4.1 状态定义

| 状态     | 说明 |
|----------|------|
| PENDING  | 待审批，刚提交的申请 |
| APPROVED | 已通过，审批人执行“通过”后 |
| REJECTED | 已驳回，审批人执行“驳回”并填写原因后 |

### 4.2 状态流转

```
  [提交申请] → PENDING
                    ↓ 审批通过
                 APPROVED
                    ↓ 审批驳回
                 REJECTED
```

- 仅当状态为 **PENDING** 时可执行“通过”或“驳回”。  
- 通过/驳回时记录审批人 ID、审批时间；驳回时必填驳回原因（reject_reason）。

### 4.3 数据校验

- 申请时：药品 ID、采购数量必填；数量应为正整数。  
- 审批时：同一申请不可重复审批（状态已非 PENDING 则拒绝）。

## 5. 处方审核逻辑

### 5.1 状态定义

| 状态      | 说明 |
|-----------|------|
| DRAFT     | 草稿，新建或未提交 |
| SUBMITTED | 已提交，待审核 |
| APPROVED  | 已通过，审核人通过 |
| REJECTED  | 已驳回，审核人驳回 |

### 5.2 状态流转

```
  [新建处方] → DRAFT
                  ↓ 提交
               SUBMITTED
                  ↓ 审核通过
               APPROVED
                  ↓ 审核驳回
               REJECTED
```

- 仅 **DRAFT** 可执行“提交”；仅 **SUBMITTED** 可执行“审核通过”或“审核驳回”。  
- 通过/驳回时记录审核意见（review_remark）、审核时间（reviewed_at）。

### 5.3 处方与明细

- 处方主表（prescription）与明细表（prescription_item）为一对多。  
- **患者病情信息**：处方主表包含字段 **patient_condition**（患者病情信息，主诉、诊断等），新建处方时可选填，列表展示摘要、详情与明细弹窗中完整展示，供审核人员参考。  
- 新建处方时先保存主表（含 patient_condition），再通过“添加明细”接口逐条保存明细（药品、剂量、途径、时长）；医生角色需具备**药品列表只读**权限，以便在添加处方药品时从药品库选择。  
- 审核时可根据患者病情与明细、药品库做剂量、配伍等校验（当前可为简单校验或提示，完整规则见需求与难点分析）。

## 6. 患者用药指导逻辑

### 6.1 输入与输出

- **输入**：用户文本（症状、用药或问题描述），长度受配置限制（如最大 500 字符），超长返回 400 及提示。  
- **输出**：系统返回建议文本 + 免责声明（“仅供参考，请遵医嘱”）；匿名用户成功响应中附带本窗口内**剩余提问次数**（remainingAsks），便于前端展示。

### 6.2 匿名限流（未登录用户）

- **适用对象**：未携带有效 JWT 或未认证的请求视为匿名。  
- **限流规则**：按**客户端 IP**（优先 X-Forwarded-For、X-Real-IP，否则 RemoteAddr）在**滑动时间窗口**内计数；默认窗口 1 小时、每 IP 最多 3 次提问。  
- **超限处理**：本窗口内已满 3 次时，拒绝本次提问，返回 HTTP 200 + 统一体 `code: 429`、`message` 为“未登录用户每 X 小时内仅可提问 Y 次，请登录后继续使用或稍后再试。”  
- **已登录用户**：不参与上述限流，提问次数不限。  
- **配置**：`guide.limit` 下可配置 `max-asks-per-window`、`window-hours`、`question-max-length`（见开发与部署说明）。

### 6.3 回复生成：规则 + AI 双模式

- **优先 AI**：若配置了 OpenAI 兼容接口（如 DeepSeek，见 `guide.ai`），则调用外部接口生成回复；调用失败或未配置时进入规则兜底。  
- **规则兜底**：关键词规则：包含“感冒”“发烧”返回对应建议；包含“过敏”返回过敏相关提示；默认返回通用用药与就医提醒。  
- 界面与接口回复中统一保留免责声明，不暴露内部错误信息。

### 6.4 责任边界

- 系统回复不替代医生诊断与处方，仅作参考；界面与接口文档中均需明确免责说明。

## 7. 系统管理逻辑

### 7.1 登录逻辑

1. 根据用户名查询用户，不存在则返回“用户名或密码错误”（不暴露是否存在该用户）。  
2. 校验账号是否启用（enabled），禁用则拒绝登录。  
3. 使用 BCrypt 校验密码，不匹配则返回“用户名或密码错误”。  
4. 成功后返回用户信息（不含密码），供前端展示或后续 JWT 生成。

### 7.2 用户管理逻辑

- **新增**：用户名不可与已有用户重复；密码存储前必须 BCrypt 加密。  
- **修改**：可修改姓名、角色、启用状态；若传入新密码则加密后更新，否则保留原密码。  
- **列表/详情**：接口返回用户信息时一律脱敏，不返回 password 字段。

### 7.3 角色与权限（RBAC 简化）

- 用户表 **role** 字段表示角色（如 ADMIN、PHARMACIST、REVIEWER 等）。  
- 前端可根据 role 控制菜单与按钮显示；后端可在 Spring Security 中按角色限制接口访问（当前版本为全部放行，便于毕设演示）。

## 8. 异常与事务

### 8.1 事务边界

- 所有写操作（入库、出库、审批、处方提交/审核、用户增改）均在 Service 层方法上使用 **@Transactional**，保证单次请求内数据一致。  
- 查询方法使用 **@Transactional(readOnly = true)** 或默认只读，减少锁竞争。

### 8.2 异常处理

- 业务校验失败（如库存不足、状态不允许操作、用户名重复）抛出 **RuntimeException** 或自定义业务异常，由全局异常处理器或前端拦截器统一转换为统一错误响应（code≠200、message 为错误说明）。

## 9. 用户角色与登录、权限流程

### 9.1 登录流程

1. 用户在前端登录页输入用户名、密码，提交至 `POST /api/auth/login`。  
2. 后端校验用户名存在、账号启用、密码（BCrypt）正确后，生成 JWT token（携带 userId、username、role），并返回 `{ token, user }`（user 中密码已脱敏）。  
3. 前端将 token 与 user 存入本地（如 localStorage），之后每次请求在请求头携带 `Authorization: Bearer <token>`。  
4. 后端 `JwtAuthFilter` 解析 token，将用户身份与角色写入 Spring Security 上下文，供接口权限判断使用。

### 9.2 角色与模块权限（中国医院操作流程）

| 角色       | 药品信息       | 库存管理 | 采购审批 | 处方审核 | 用药指导 | 系统管理 |
|------------|----------------|----------|----------|----------|----------|----------|
| 管理员     | ✓              | ✓        | ✓        | ✓        | ✓        | ✓        |
| 药师       | ✓              | ✓        | —        | —        | ✓        | —        |
| 采购员     | 只读（选药品） | ✓        | ✓        | —        | ✓        | —        |
| 审核员     | 只读（选药品） | —        | ✓        | ✓        | ✓        | —        |
| 医生       | **只读（处方选药）** | —    | —        | ✓        | ✓        | —        |
| 护士       | —              | ✓        | —        | 只读（查询） | ✓      | —        |

- 后端：各模块对应 URL 在 `SecurityConfig` 中按角色限制（hasRole / hasAnyRole）；**药品 GET 接口**允许 ADMIN、PHARMACIST、PURCHASER、REVIEWER、**DOCTOR**，以便医生在处方审核页添加药品时加载药品列表；未认证或角色不符返回 401/403。  
- 前端：路由守卫未登录跳转登录页；按角色仅展示有权限的菜单与路由，避免越权访问。

### 9.3 登出与 token 失效

- 用户点击退出时，前端清除本地 token 与用户信息并跳转登录页。  
- 若 token 过期或非法，后端返回 401/403，前端响应拦截器清除本地登录态，下次路由跳转时进入登录页。
